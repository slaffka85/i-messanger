# План реализации: Чат, Файлы и Google OAuth2

В связи с тем, что нам требуется сохранение истории сообщений и авторизация пользователей (чтобы мы знали, кому принадлежит история), архитектура проекта будет расширена.

---

## Предлагаемые изменения

### 1. Авторизация (Google OAuth2)
Да, локально прикрутить авторизацию через Google можно (и это очень удобно для аутентификации).

#### [Бэкенд - Spring Boot]
- **Зависимости**: Добавим `spring-boot-starter-oauth2-client` и `spring-boot-starter-security`.
- **Настройка Security**: Создадим конфиг `SecurityConfig.java`. По умолчанию все эндпоинты будут защищены, кроме статики и логина. Будет включен OAuth2 Login. После успешной авторизации через Google пользователь будет сохранен в нашу локальную БД.
- **Переменные окружения**: В `application.yml` добавим заглушки под `GOOGLE_CLIENT_ID` и `GOOGLE_CLIENT_SECRET`. Вам потребуется создать приложение в Google Cloud Console и вставить туда эти ключи.

#### [Фронтенд - React]
- **Редирект на логин**: Если пользователь не авторизован (проверка через эндпоинт `/api/users/me`), его будет редиректить на страницу с кнопкой "Войти через Google" (`<a href="http://localhost:8080/oauth2/authorization/google">`).
- **Идентификация WebSocket**: Так как авторизация будет работать на базе сессий (Cookies), при WebSocket-подключении сервер будет автоматически понимать, какой пользователь подключается, и нам больше не придётся использовать временные ID.

### 2. База Данных (PostgreSQL)

#### [Миграции Liquibase / NEW]
Создадим скрипты для таблих:
- Таблица `users`: `id`, `email`, `name`, `avatar_url`, `created_at`.
- Таблица `messages`: `id`, `sender_id` (FK), `recipient_id` (FK), `text`, `file_id`, `created_at`.

#### [JPA Entity & Repositories / NEW]
- Создадим Entity-классы `User` и `Message`.
- Создадим Spring Data репозитории `UserRepository` и `MessageRepository`.

### 3. Функционал обмена сообщениями и файлами

#### [REST API для истории и файлов]
- `GET /api/messages/{userId}`: получить историю переписки между текущим пользователем и другим `userId`.
- `POST /api/files/upload`: загрузка файла на сервер, возвращает `fileId`.
- `GET /api/files/{fileId}`: скачивание файла.

#### [WebSocket SignalingHandler]
- **Авторизация**: Вместо временных `clientId`, мы будем брать `User Principal` прямо из сессии.
- **Роутинг чата (chat-message)**: Когда поступает сообщение, сервер будет сначала сохранять его в таблицу `messages` через `MessageRepository`, а уже затем ретранслировать получателю через WebSocket.

#### [Фронтенд - Компонент Чата]
- Создание компонента `Chat.jsx`, который будет подгружать историю сообщений при открытии.
- Интеграция чата в лобби (со списком авторизованных пользователей) и в окно звонка.

## User Review Required

> [!IMPORTANT]
> **Настройка Google Cloud Console**
> Для работы авторизации вам необходимо будет зайти в [Google Cloud Console](https://console.cloud.google.com/), создать проект, настроить экран согласия (OAuth Consent Screen) и создать **Учетные данные (OAuth 2.0 Client IDs)**. В качестве "Разрешенных URI перенаправления" надо будет указать `http://localhost:8080/login/oauth2/code/google`. \
> **Сможем ли мы заняться этим вместе, когда я подготовлю код?**

> [!WARNING]
> **Архитектура сессий и CORS**
> Так как бэкенд (8080) и фронтенд (5173 - vite) работают на разных портах, передача cookies (сессии авторизации) требует правильной настройки CORS (`allowCredentials(true)`). Мы настроим это в Spring Security.

## Verification Plan

### Manual Verification
1. Запуск проекта. Попытка открыть интерфейс перенаправляет на страницу авторизации.
2. Клик по "Войти через Google", авторизация, редирект обратно в приложение.
3. В базе данных (в DBeaver/DataGrip) появляется запись о новом пользователе в таблице `users`.
4. В лобби мы видим список всех когда-либо зарегистрированных пользователей (вместо временных ID).
5. Пользователь A отправляет сообщение пользователю B. Сообщение сохраняется в БД `messages`. Если B онлайн, он мгновенно получает его через WebSocket.
6. При обновлении страницы у A или B история сообщений успешно подгружается из REST API.
